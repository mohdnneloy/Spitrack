    1. Introduction

    1.1 Pupose

    Spitrack is a web based search engine that uses crawling and scraping strategies to search some specific content out of a given cluster of URLs (Unified Resource Locator). The application is mainly focused on simplifying research strategies for researchers. In addition to the web-app, a mobile-app is a complementary option to facilitate the searching capability through portable methods. The search can be done over some scrapped data on a central server that has been retrieved from a given set of clusters created by a user.

    1.4 Product Scope

    Spitrack is focused on the simplification of research. It gives the users the ability to form clusters of URLs with some specified crawling and scraping strategies. A researcher who is focused on an individual research can use the URL links of his preference to search some content only within those links and the ones that are present inside it. Users within this system will have the ability to create clusters and search within the created clusters. Also, the admins will have the ability to add crawling strategies to the system by just installing them as plugins. They can later on activate or deactivate the crawling strategy for the users of this system. The main objective of this system is to provide a better and easier procedure to search content for an optimal research document. This SRS will provide the required information for the development of the complete or partial component of the whole system.

    2. Overall Description

    2.1 Product Perspective

    The product Spitrack is about simplifying search methodologies. It will help users to search to the point, without going through the hassle of searching multiple web pages. Users are going to create and use search clusters.This product is different from regular search engines in respect of the scope of searching, Spitrack will search on the created clusters, whereas others will search through the whole web. This is a completely new system that doesn't have any dependencies on subsystems of the super-systems.

    2.6 User Documentation

    Spitrack is a web based research platform for researchers where they can simplify their content by performing their research only within some specified boundary of content. Research is mostly based on URLs today as most of the contents are available on the internet. Web search engines will provide you with a lot of content that you might not need for your research. Spitrack gives you the facility to search over a set of links that you prefer. In addition, it also collects all the PDF files or Word Files or any other textual files within the link and gives the opportunity to do your research on authentic content found with the cluster of links that you provide. Last but not the least, you also have the ability to download that file from where you have done your research.

    3. External Interface Requirements

    3.1 User Interfaces

    The software interfaces for both the web-app and mobile app are required to follow the Model-View-Controller model for the acceptance of an action by the user, model data on the database and rendering the retrieved data on the user terminal. The front-end interface should be able to send HTTP requests to the back-end interface to perform some CRUD operations on the database through RESTful APIs.

    The user interfaces can be divided into the following panels:

    • User Panel
    • Admin Panel

    The user panel mainly shows the user dashboard tab, cluster formation tab, search tab and the profile tab. The dashboard shows all the recent searches and clusters formed by the user. Through the dashboard users can perform their previous searches and access the premade-clusters. Using the cluster formation tab the users can create new clusters by providing some URLs and select the crawling strategy of their choice along with a particular depth up to which the data needs to be scrapped. The profile tab shows the basic information of an individual user. The search tab allows the user to select a premade cluster and perform a search within that cluster using a search field. Once a search operation is complete, all the optimal results are displayed on the screen below the search field.

The installed plug-in tab allows the admin to see the installed plugins on its system and make those plugins active or inactive for its users.

    3.3 Software Interfaces

    This product is built on the basis of python framework “Django” and the front-end is completely developed on Javascript based library React.     4. System Features

    This section has all the possible system features for Spitrack Version 1.0. The basics of the feature requirements are included in this part, and it might be changed based on the updates in the later version.

    4.1 User Sign Up With or Without Google Authentication

    A new user to the system, enters his or her details, and creates a new user account on the system. He or she is given a new user ID. New users can also sign up into the system with their Google account directly.

    4.1.1 Description and Priority

    This feature is for the new users of this system. To sign up into the system a new user needs to insert his/her first name and last name, email and a password of their choice. Once their new account is created their details will be saved in the server and they will get direct access into their dashboard. Just in case a user wants to go for a faster sign up he/she can use the “Log In with Google” option to create his new account and “sign in” into his/her dashboard after his Google Account has been authenticated. This feature has a priority level of 9. The benefit of this feature is that no unauthenticated user can access the system, and users of this system will be safe from theft. Without an account a user will not be able to access the system.

    4.1.2 Stimulus/Response Sequences

    Sign Up without Google

    • User goes to sign up page
    • Enter his/her details (first name, last name, email and password)
    • Submit the form
    • New User account is created
    • User is directed to his/her dashboard

    4.1.3 Functional Requirements

    REQ-1: A database is required on the server side to store all the information of a new user. If no database is created prior to any sign up form submission the system will send an error of “New Account could not be created due to server issues!”

    REQ-2: All the input fields need to be filled completely before the form is submitted.

    4.2 Create Cluster of URLS or COUs

    An existing user creates a URL cluster by inserting the URLs of his/her choice and applies some crawling strategies already available on the system for the new cluster. Along with this they also set the depth of the URL level up to which they want to search and press the “Create Cluster Button”.

    4.2.1 Description and Priority

    This feature allows an existing user who already has an account on the system to create a new COU. The COU is easy to create. The user needs to insert the main or root URLs on which they want to make their research. Up next they have to select some crawling strategy available on the system like “PDFs, TXT, MS Word” etc., basically the files that they want to search for in the links. Last but not the least, they need to provide the level of depth into the URLs up to which they want to search for the files they require based on the crawling strategies selected.

    4.2.2 Stimulus/Response Sequences

    • User goes in the clusters tab on the side navigation bar
    • The “Cluster site” clicks on the “Create Cluster Button”, which will take them to the create cluster page
    • Users enter the URLs on which they want to do their research. Number of links is limited to 5.
    • User selects the crawling strategy of their choice based on their requirement using the checkbox options
    • User enters a depth for the crawl within the URLs provided. Maximum depth can be 10.
    • User clicks on “Create Cluster” to make the new COU
    • The system then shows a message “You will be notified through email when the cluster is ready”

    4.2.3 Functional Requirements

    REQ-1: Database needs to be ready for indexing of content

    REQ-2: Web Crawling and Web Scraper libraries are required for data crawling and indexing.

    4.3 View or Delete clusters already created

    An existing user of this system can view his/her created clusters.

    4.3.1 Description and Priority

    This feature allows the user to view the clusters they have already created and also gives them the option to delete a cluster out of the ones that are already available.

    4.3.2 Stimulus/Response Sequences

    • User goes on the “Clusters Tab”
    • All the clusters are shown on screen along with the URLs, crawling strategies and depth
    • User can delete a cluster by just clicking the delete button below an individual cluster
    • If user clicks on delete button
    • System removes the cluster from the clusters tab
    • System removes the data related to the deleted cluster from the server database

    4.3.3 Functional Requirements

    REQ-1: Database should be automated through the ORM to perform required migrations

    4.4 Perform Search Operation

    Existing Users can perform a search over the clusters that they have created. A search can be done easily by typing the keywords in the “Search Box” and selecting the COUs and last pressing the search button.

    4.4.1 Description and Priority

    This feature allows the user to perform some specific search on the COUs that they have created. They can perform the search over the COUs by inserting their required keywords in the search field, select the COU on which the search needs to be done. Once the user clicks on the search button, a set of possible results are shown below the search bar. Each search result shows the link where the keyword is found along with its root link and parent link with their respective depths. Once the results are available the user can directly download the file from the URL or access the page if any.

    4.4.2 Stimulus/Response Sequences

    • User inputs the keywords in the search box
    • User selects the COU
    • User clicks on search button
    • System provides all possible results

    4.4.3 Functional Requirements

    REQ-1: At Least one COU needs to be created before any search

    REQ-2: Database in the server needs to be online to perform the search

    4.5 Add New Crawling Strategy through Plug-In

    Admins will be able install a crawling strategy though plugin.

    4.5.1 Description and Priority

    Admins will be able to add “install” a new crawling strategy into the system from the available plugins in the “Install Plugin Page”. Admin can move into the “Install Plug In Page” and install a crawling strategy plugin. Later on he/she moves into the installed Plugins tab, and then they have the option to either activate or deactivate the installed plugin. If the plugin is deactivated the users of this system will not be able to use that crawling strategy plug in.

    4.5.2 Stimulus/Response Sequences

    • Admin goes to the “Install Plugin” page
    • Selects a plugin of their choice and clicks on Install
    • The system installs the plugin
    • Admin is taken automatically to the “Installed plugin tab”
    • Admin selects “Activate Plugin” or “Deactivate Plugin” accordingly.

    4.5.3 Functional Requirements

    REQ-1: Admin account needs to be created

    REQ-2: Plugins must be available in the “Plugins Page”

    REQ-3: Admin must be authorized

    Appendix A: Glossary

    • ORM (Object Relational Mapping): Object-relational mapping (ORM) is a mechanism that makes it possible to address, access and manipulate objects without having to consider how those objects relate to their data sources.

    • DBMS (Database Management System): Database management system will store, retrieve, define, and manage data in a database.

    • API (Application Programming Interface): Application programming Interface is a software intermediary that allows two applications to talk to each other.

    • REST (Representational State Transfer): A RESTful API is an architectural style for an application program interface (API) that uses HTTP requests to access and use data.

    • SRS (Software Requirements Specification): A software requirements specification (SRS) is a document that describes what the software will do and how it will be expected to perform. It also describes the functionality the product needs to fulfill all stakeholders (business, users) needs.

    • REACT (Front-End Javascript Library): REST is a JavaScript library for building user interfaces

    • DJANGO (Back-End Python Framework): Django is an advanced Web framework written in Python that makes use of the model view controller (MVC) architectural pattern.

    • SQL (Structured Query Language): SQL (Structured Query Language) is a standardized programming language that's used to manage relational database

    • COU (Cluster of URLs): Cluster of URLs is a list of URLs in which the list will be provided by the user.

    • WCS (Web Crawling Strategy): Web crawling is the process of indexing data on web pages by using a program or automated script.

    • PII (Plugin Installer): Plugin Installer adds new functions to a host program without altering the host program itself.

    • DOM (Document Object Model): Document Object Model is a platform and language-neutral interface that allows programs and scripts to dynamically access and update the content, structure, and style of a document.

    • CRUD (Create Read Update and Delete): CRUD is an acronym that refers to the four functions that are considered necessary to implement a persistent storage application: create, read, update and delete.
